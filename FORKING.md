# Forking, Independent Clients, and Community Projects

PODS is an open-source project, and forks, alternative clients, tooling, experiments, and community projects are encouraged.

If you build something cool with PODS, I genuinely want to see it.

This document exists to clarify the difference between:
- community-driven projects and forks, which are welcome
- impersonation or misleading branding, which is not

---

# What You Are Allowed To Do

You are welcome to:

- Fork the project
- Modify the source code
- Create independent clients
- Create independent servers
- Build replay tools, analytics tools, overlays, bots, accessibility tools, or protocol implementations
- Monetize your own independent implementations
- Distribute modified builds
- Experiment with gameplay changes or protocol changes

Commercial projects are allowed under the terms of the AGPLv3 license, provided you comply with the license requirements.

---

# Naming and Branding Guidelines

Please make it clear when your project is unofficial.

Good examples:

- "DERP — An Independent PODS Client"
- "ORBITAL — Compatible with PODS"
- "PODS Protocol Replay Viewer"
- "Unofficial PODS Spectator Client"

Please avoid naming that implies official endorsement or identity.

Examples of names that are NOT okay:

- "Official PODS"
- "PODS Remastered"
- "PODS 2"
- "Eldenbrig PODS"
- anything using official Eldenbrig Games branding in a misleading way

---

# Logos, Art, Audio, and Branding

Unless explicitly stated otherwise, the AGPLv3 license applies to source code only.

Official branding and creative assets may be licensed separately, including but not limited to:

- logos
- icons
- music
- sound effects
- visual assets
- lore text
- promotional artwork

Do not use official branding or assets in ways that imply your fork or project is the official release.

---

# Compatibility References

You are welcome to reference PODS for compatibility purposes.

Examples:

- "Compatible with PODS servers"
- "Built for the PODS protocol"
- "Third-party PODS client"

Please do not present your project as the canonical or official PODS release unless you are explicitly authorized to do so.

---

# Compatibility and Support Expectations

Forks and derivative projects are encouraged to clearly communicate:

- which protocol version they support
- whether they are compatible with official PODS infrastructure
- whether gameplay or protocol behavior has changed
- where users should report bugs or compatibility issues

Where practical, forks should expose:

- project name
- repository or homepage
- issue tracker URL
- compatibility information

within the application UI, launcher, server console output, documentation, or distribution page.

Examples:

Compatible with:
- PODS Protocol v1.2
- Official Servers: Yes

Project:
- DERP Client
- https://github.com/example/derp-client
- https://github.com/example/derp-client/issues

Forks are free to modify gameplay systems, networking behavior, physics models, rendering systems, matchmaking, replay formats, or protocol behavior.

However, forks that intentionally diverge from official PODS behavior are encouraged to communicate this clearly to users.

This helps reduce confusion for players and prevents issues relating to unofficial implementations from being incorrectly reported to the official PODS repositories.

---

# Community Philosophy

PODS is intended to support experimentation, creativity, accessibility, and community-driven development.

Alternative clients, weird rendering experiments, accessibility-focused implementations, replay systems, bots, overlays, protocol experiments, and total gameplay conversions are all part of the kind of ecosystem this project hopes to encourage.

Forks are welcome.

Impersonation is not.
